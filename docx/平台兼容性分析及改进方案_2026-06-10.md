# 平台兼容性分析及改进方案

**日期**: 2026-06-10
**版本**: v0.5A
**分析目标**: 保证项目兼容 yt-dlp 支持的 1800+ 视频平台

---

## 一、背景

yt-dlp 原生支持 1800+ 视频/音频平台的解析与下载。项目当前通过 `ytdlp_bridge.py` 桥接模块封装 yt-dlp，对 **Bilibili / 抖音 / 快手 / YouTube** 四个平台做了针对性优化（请求头、提取器参数、格式备注标准化），其他平台透传至 yt-dlp 原生 extractor 处理。

经代码审查，**项目不存在硬编码的 URL 限制或域名白名单**，理论上任何 yt-dlp 支持的链接均可传入。但仍存在 6 个方面可优化，以提升跨平台兼容性。

---

## 二、各模块限制分析

### 2.1 Java 前端层（HomeFragment.java）

| 检查点 | 结论 |
|--------|------|
| URL 输入校验 | ✅ 仅检查空字符串，无格式/域名过滤 |
| 平台判断 | ❌ 无平台判断，直接透传至 Python |
| Cookie 传递 | ⚠️ 仅传递 Bilibili Cookie 文件 |

### 2.2 Python 桥接层（ytdlp_bridge.py）

| 函数 | 已知平台 | 未知平台行为 |
|------|----------|-------------|
| `_detect_platform()` | bilibili/douyin/kuaishou/youtube | 返回 "unknown" |
| `_get_common_opts()` | 设置对应 Referer/Origin | 不设请求头（安全） |
| `extractVideoInfo()` | 已知平台额外参数 + 重试逻辑 | 直接调用 yt-dlp |
| `_normalize_platform_formats()` | douyin/kuaishou 标准化 format_note | 跳过标准化 |

### 2.3 依赖层（requirements.txt / build.gradle.kts）

| 依赖 | 现状 | 问题 |
|------|------|------|
| yt-dlp | 裸包，无 extra | 缺少 `curl-cffi`，TLS 指纹模拟不可用 |

## 三、改进项详述

### 3.1 【P0】更新 User-Agent

**位置**: [ytdlp_bridge.py:115](file:///d:/COMPUTER/Android/GrayTools/Files/app/src/main/python/ytdlp_bridge.py#L115-L119)

**现状**:
```python
"User-Agent": (
    "Mozilla/5.0 (Windows NT 10.0; Win64; x64) "
    "AppleWebKit/537.36 (KHTML, like Gecko) "
    "Chrome/130.0.0.0 Safari/537.36"
)
```

Chrome/130 发布于 2024 年 10 月，距今已约 20 个月（当前 Chrome 约 ~150+）。部分对 UA 敏感的平台可能开始拦截。

**修改方案**: 从 `UserAgent` 常量映射表中选取最新的 Chrome 版本号替代 130。

---

### 3.2 【P0】安装 curl_cffi + 启用 impersonate

**位置**: [ytdlp_bridge.py:147-152](file:///d:/COMPUTER/Android/GrayTools/Files/app/src/main/python/ytdlp_bridge.py#L147-L152)

**现状**:
```python
if platform in ("douyin", "kuaishou"):
    opts["extractor_args"] = opts.get("extractor_args", {})
    # 空操作，未设置 impersonate
```

**问题**: 以下平台依赖 TLS 指纹模拟（impersonate）才能正常访问：
- TikTok / 抖音 / 快手（含海外版）
- Instagram（Reels / 视频）
- Twitter / X（视频）
- 部分流媒体平台

**修改方案**:
1. 将 requirements.txt 中的 `yt-dlp` 改为 `yt-dlp[curl-cffi]`
2. 在 `_get_common_opts()` 中为相关平台添加 `"impersonate": True` 或指定目标浏览器

**位置**: 
- [HomeFragment.java:770](file:///d:/COMPUTER/Android/GrayTools/Files/app/src/main/java/com/example/grayvideodl/ui/home/HomeFragment.java#L770)
- [DownloadFragment.java:573](file:///d:/COMPUTER/Android/GrayTools/Files/app/src/main/java/com/example/grayvideodl/ui/download/DownloadFragment.java#L573)

**现状**:

**问题**: 需要登录才能访问内容的平台（Twitter/X、Instagram、微博等）无法使用对应的 Cookie 注入。

## 五、技术风险与注意事项

1. **impersonate 对 Android 的影响**: curl_cffi 的 TLS 指纹模拟在 ARM64 架构（Android 设备）上需要原生库支持，需在 Chaquopy 环境中验证可用性。
2. **多平台 Cookie 管理复杂度**: 不同平台的 Cookie 格式（Netscape 格式通用，但字段含义有差异）和过期策略不同，需统一规范。
3. **兼容性回退**: 所有已知平台特殊处理（请求头、impersonate、extractor_args）应在失败时自动降级为通用模式，保证不降低现有成功率。
4. **extractVideoInfo 重试逻辑扩展**: 当前重试逻辑仅处理 Cookie 导致的 412/403 错误。启用 impersonate 后，可能需要扩展对 TLS 相关错误的检测与回退。
